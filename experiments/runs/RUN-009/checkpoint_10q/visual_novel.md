# Визуализация новеллы через AI: рабочий бриф

## Контекст

Хочу визуализировать текстовую новеллу так, чтобы это было **приятно читать/смотреть**, а не просто получать случайные несвязные картинки.

У меня есть:
- **Tesla V100 32 GB**
- **RTX 5060 Ti 16 GB**
- инженерный опыт в LLM / orchestration / backend
- желание получить **первый вменяемый результат за пару вечеров**, а не строить полгода идеальный комбайн

Мой профиль по навыкам ближе не к “хочу кнопку magic”, а к:
- backend orchestration
- pipeline design
- tool routing
- data contracts
- методологическая работа
- applied LLM / research mindset

---

# Главный вывод

## Что делать

Делать **не full-auto аниме**, а **motion comic / illustrated novel pipeline**:

`глава -> сцены -> шоты -> ключевые кадры -> локальные анимации/переходы -> опционально озвучка`

Это даёт:
- смотрибельный результат
- меньшую цену ошибок
- меньше ручной боли
- лучший fit под мой GPU-сетап
- больше контроля над консистентностью персонажей и сцен

---

## Чего не делать

Не ставить целью сразу:
- полностью автоматическое “новелла -> готовая серия”
- длинное видео на каждую сцену
- идеальную авто-раскадровку без редакторского прохода
- один “магический” монолитный пайплайн, который делает всё сам

Почему:
- качество ломается именно на этапе **декомпозиции текста в киноязык**
- персонажи начинают “плыть”
- сцены распадаются по стилю
- длинное видео резко увеличивает стоимость времени и ошибок
- самый хрупкий участок — не рендер, а **story interpretation**

---

# Рекомендуемая стратегия

## Лучшая форма MVP

Сделать систему, которая превращает главу в:

1. список сцен  
2. список шотов внутри сцены  
3. JSON-манифест  
4. набор ключевых кадров  
5. по желанию — лёгкие анимированные вставки  
6. финальную склейку в формате:
   - PDF/CBZ-like comic
   - vertical storyboard
   - short motion comic video

У меня уже есть инженерный бэкграунд, значит мне выгоднее не искать “волшебную коробку”, а собрать **тонкий оркестратор поверх существующих UI и движков**.

Мой прошлый проект (`rag_app`) показывает, что я уже умею:
- строить сервисы
- держать pipeline в контрактах
- делать planner/tool-runner architecture
- собирать multi-step workflows
- думать в терминах evaluation / methodology

Значит лучший путь для меня:
- не рисовать всё руками в Comfy бесконечно
- не надеяться на 100% коробочное решение
- а сделать **свой orchestration layer**, но поверх уже готовых инструментов

# Конкретная рекомендация по стеку

## Вариант, который я бы рекомендовал как основной

### Оркестратор
Свой backend на FastAPI / Python.

Роль:
- загрузка главы
- разбиение на сцены
- генерация shot plan
- хранение канона персонажей
- кэш и повторяемость

- сборка финального результата

#### 3. StoryDiffusion
Использовать для **консистентности персонажей** между сценами и кадрами.

Подходит для:

#### 4. WanGP
Использовать как удобный UI/слой для коротких video/i2v задач.

Подходит для:
- короткие clip generation

#### 5. TTS
Для озвучки:
- Kokoro — если нужен лёгкий и понятный вариант

#### 6. Финальная сборка
FFmpeg + свой Python assembler.

# Что не брать как основу

## Не делать ставку на один “всё-в-одном” продукт

Да, есть красивые пайплайны вроде:
- AIComicBuilder
- LumenX

Но их лучше использовать как:
- референс UX
- источник методологии
- место подсмотреть pipeline stages

А не как основу своего локального продакшна, потому что:
- часто завязаны на внешние API
- хуже подходят под локальный heterogeneous hardware setup
- меньше контроля
- сложнее встраивать в собственную инженерную логику

# Правильная архитектура под мои 2 GPU

## Принцип
Не смешивать всё в один процесс.

Делать:
- **разные backend-сервисы**
- **жёсткая привязка к своей GPU**
- общий orchestration layer сверху

---

### V100 32 GB

# Идеальная схема сервисов

## Сервисы

### 1. `novel-orchestrator`
Мой backend.

Функции:
- ingest chapter
- scene extraction
- character memory
- shot planning
- dispatch jobs
- collect outputs
- produce final package

---

### 2. `llm-backend-5060`
Отдельный LLM endpoint на 5060 Ti.

Функции:
- chapter -> scenes
- scenes -> shots
- shot prompt drafting
- negative prompt drafting
- continuity checks
- dialogue packaging

---

### 3. `render-worker-v100`
Основной image worker.

Функции:
- keyframes
- character references
- scene renders
- correction passes
- upscale

---

### 4. `video-worker`
Либо на V100, либо на 5060 Ti, в зависимости от модели и задачи.

Функции:
- short i2v
- panel animation
- transition clips

---

### 5. `audio-worker`
TTS / narration / voice cleanup

---

### 6. `assembler`
Склейка:
- frames
- subtitles
- voice
- music
- transitions
- экспорт

Почему он вмешался именно в тот момент? Потому что в контексте сцены Хирата уже не просто “апатичный”, а ведёт себя некрасиво: грубо отталкивает Мии-тян, доводит её до слёз и срывается. В тексте Коэндзи цепляется именно за это: он упрекает Хирату за то, что тот заставляет девушку плакать, требует, чтобы тот хотя бы протянул ей руку, а затем сам уводит Мии-тян. То есть триггером для Коэндзи был не “интерес класса”, а сочетание двух вещей: Хирата стал для него раздражающим “уродливым” зрелищем, и при этом плохо обошёлся с девушкой.

Поэтому правильнее читать эту сцену так: Коэндзи не спасает класс из альтруизма, а действует по своей прихоти и по своему вкусу к ситуации. Просто его личное вмешательство случайно оказалось полезным: оно вытолкнуло Хирату из пассивного самоуничтожения и создало момент, после которого Аянокодзи уже смог отдельно найти Хирату и начать тот самый разговор о его прошлом и вине. Иначе говоря, Коэндзи здесь — скорее катализатор, а не человек, который “решил стать командным игроком”.

Ещё в Year 1 Volume 10 Аянокодзи замечает, что во время истории с Ямаути Коэндзи постоянно провоцировал его так, чтобы всю ненависть Ямаути после исключения стянуло на самого Коэндзи, а не на весь класс. Сам Коэндзи, как обычно, отмахивается и не признаёт “благородный” мотив. То есть паттерн у него уже был: он не становится хорошим одноклассником, просто иногда его личные импульсы совпадают с выгодой класса.

То есть формула у него ближе не к “я хочу всем вредить”, а к “никто не имеет права распоряжаться мной, но если я сам считаю что-то достойным — я вмешаюсь”. Из-за этого со стороны он кажется хаотичным: сегодня он мешает, завтра спасает, послезавтра смеётся и уходит. Но по тексту это не хаос, а последовательность по его собственным правилам, просто эти правила не совпадают с нормальной школьной моралью и логикой класса. В фанатских обсуждениях его часто именно так и читают: не как внезапно “исправившегося”, а как персонажа, который всегда действует только по собственной воле и иногда его воля случайно совпадает с выгодой других.

Сравнение с Аянокодзи у тебя тоже частично попадает в цель, но между ними важная разница. Оба скрывают масштаб своих возможностей, однако Аянокодзи обычно сдерживает себя и растворяется, а Коэндзи, наоборот, демонстративно выпячивает себя. Аянокодзи работает через контроль и маскировку, Коэндзи — через отказ подчиняться любым рамкам, кроме своих. Поэтому Коэндзи выглядит как “Аянокодзи без поводка” только на поверхности; по сути он другой типаж — не скрытый инструменталист, а человек, который обожествляет собственную автономию.

Самый сильный контраргумент против идеи “в глубине души он всё-таки secretly за класс” — это его поведение дальше. Мой итог: да, Коэндзи точно “не весь на поверхности”. Но то, что он утаивает, похоже не на “тайную любовь к классу”, а на личную систему ценностей, в которой есть место уважению к искренности, презрению к лицемерию и отказу быть чьим-либо инструментом. Поэтому он не полный эгоист в примитивном смысле, но и не скрытый добряк. Он ближе к персонажу, который помогает только тогда, когда это соответствует его эстетике, гордости и личному кодексу.

Да — и это как раз правильнее, чем пытаться запихнуть всё в один процесс. Для разных по типу GPU самый надёжный паттерн сейчас — разделение по задачам и по сервисам: один backend жёстко сидит на V100, другой — на 5060 Ti, а сверху у тебя общий UI или роутер. Даже в vLLM для multi-GPU прямо рекомендуют внутри одного инстанса использовать одинаковые GPU, а для heterogeneous-сценария запускать отдельные инстансы и управлять ими независимо. Привязка контейнера к конкретной карте штатно делается через device_ids в Docker Compose.

Для картинок и видео самый практичный вариант с UI — это StableSwarmUI или ComfyUI-Distributed. У StableSwarmUI можно добавить несколько self-start backends и выставить каждому свой GPU_ID; очередь генераций потом сама раскидывается по этим backend’ам, и то же самое можно делать даже на нескольких машинах. У ComfyUI-Distributed есть режим master/worker: master координирует очередь и workflow, а worker’ы делают только GPU-вычисления; причём master можно оставить вообще orchestrator-only, без локального рендера. Это очень хорошо ложится на схему “V100 — тяжёлый рендер/батчи, 5060 Ti — быстрые или отдельные задачи”.

Для LLM-части хорошая связка — Open WebUI + отдельные OpenAI-compatible endpoints. Open WebUI официально умеет подключаться к любому backend’у, который говорит на OpenAI-compatible API, так что ты можешь держать, например, vllm или другой сервер на 5060 Ti для разбора глав, сцен, промптов и диалогов, а рядом — другой endpoint для второго класса задач. Если захочешь уже не просто ручной выбор модели, а программируемый роутинг, тогда подходит Ray Serve: у него есть явное выделение GPU/ресурсов на каждый deployment, но это уже более “инженерный” уровень, а не quickest-win.

По распределению ролей я бы делал так. RTX 5060 Ti 16 GB — это более новый Blackwell; V100 32 GB — это более старый Volta, но с большим объёмом VRAM. В документации ComfyUI отдельно отмечено, что bf16 имеет смысл на новых картах, которые его поддерживают, а V100 выигрывает прежде всего объёмом памяти. Поэтому всё, что любит новую архитектуру, низкую задержку и современные precision-режимы, я бы сажал на 5060 Ti: scene extraction, prompt refinement, диалоговый LLM, TTS, короткие служебные video/i2v-задачи. Всё, что упирается в VRAM, размеры батча, высокое разрешение, SDXL/upscale, пачки кадров, логичнее отдавать на V100 32 GB.

То есть базовая архитектура под твою задачу выглядит так:

Open WebUI без GPU как общий чат/оркестратор для текстовой части.
LLM backend на 5060 Ti для “глава → сцены → shot list → промпты”. vLLM лучше держать отдельным инстансом на этой карте, а не мешать его с V100 в одном процессе.
SwarmUI или ComfyUI master как image/video-оркестратор.
Worker на V100 для основного рендера панелей, img2img, upscale, frame batches.
Опциональный второй worker на 5060 Ti для лёгких задач или когда LLM простаивает. SwarmUI как раз умеет раскидывать очередь между backend’ами.
FFmpeg/сборка можно оставить на CPU. Это отдельный этап и не обязан жить в том же процессе, что генерация.

Я бы не делал ставку на “одно тяжёлое поколение внутри одного адресного пространства на обе карты сразу” как на первую версию системы. Для heterogeneous-набора вроде V100 + 5060 Ti гораздо стабильнее и удобнее task-level parallelism: отдельные очереди, отдельные порты, отдельные контейнеры. А уже потом, если захочешь, можно тестировать более умные схемы поверх master/worker. На практике это и проще дебажить, и лучше переживает несовместимости по precision, CUDA и model kernels.

Потому что твой сильный навык, судя по репозиторию, — не рисовать пайплайны вручную часами, а строить контролирующий слой с контрактами, кэшем, стримингом и инструментами. Я бы буквально переиспользовал архитектурную идею твоего rag_app: Query Planner превращается в Scene Planner, ReAct ToolRunner — в Shot Compiler, а ingest/knowledge слой — в canon memory по персонажам, локациям и уже сгенерированным визуальным якорям.

Самая важная мысль здесь: не генерируй сразу по сырому тексту главы. Генерируй через такой промежуточный chapter_manifest.json. Тогда у тебя появляется контроль, кэш, повторяемость, ручная правка и нормальная отладка. Это как раз очень “по-твоему” инженерно.

По GPU я бы делал так. V100 32 GB — основной render worker для keyframes, character sheets, SDXL/Flux-уровня картинок, апскейла и батчей. 5060 Ti 16 GB — служебный и быстрый worker для scene parsing, prompt refinement, TTS и коротких i2v/видео-экспериментов через WanGP. Для heterogeneous-набора это нормальный путь: ComfyUI-Distributed прямо пишет, что он не объединяет VRAM, но позволяет использовать разные GPU и даже гонять master в orchestrator-only режиме, а работу раздавать worker’ам.

Что я бы не советовал:
не строить всё вокруг Open WebUI, потому что у тебя уже есть backend-скелет сильнее, чем нужно для этой задачи;
не гнаться сразу за полноценным long video;
не делать “полностью автоматический правильный сториборд”, потому что именно здесь сейчас больше всего ломающейся магии. Свежие инструменты уже хороши в импорте, shot breakdown и ручной правке, но не настолько хороши, чтобы полностью заменить editor pass.

Если сформулировать совсем жёстко: для тебя лучший путь — не искать готовый локальный комбайн, а собрать “novel compiler + storyboard UI + render workers”. Из коробок использовать не “всё сразу”, а только те, что действительно сильны в своей части:
AIComicBuilder — как референс UX и методологии,
StoryboardUI2 — как shot-level редактор,
SwarmUI/Comfy — как генеративный двигатель,
StoryDiffusion — как консистентность персонажей,
WanGP — как лёгкий видео-слой.

Мой итог: тебе не нужна “коробка для новичка”; тебе нужен полуготовый production stack, где ты дописываешь только оркестратор и контракты данных. Это ровно тот случай, где твой existing skillset уже даёт преимущество.

# Визуализация новеллы через AI: рабочий бриф

## Контекст

Хочу визуализировать текстовую новеллу так, чтобы это было **приятно читать/смотреть**, а не просто получать случайные несвязные картинки.

У меня есть:
- **Tesla V100 32 GB**
- **RTX 5060 Ti 16 GB**
- инженерный опыт в LLM / orchestration / backend
- желание получить **первый вменяемый результат за пару вечеров**, а не строить полгода идеальный комбайн

Мой профиль по навыкам ближе не к “хочу кнопку magic”, а к:
- backend orchestration
- pipeline design
- tool routing
- data contracts
- методологическая работа
- applied LLM / research mindset

---

# Главный вывод

## Что делать

Делать **не full-auto аниме**, а **motion comic / illustrated novel pipeline**:

`глава -> сцены -> шоты -> ключевые кадры -> локальные анимации/переходы -> опционально озвучка`

Это даёт:
- смотрибельный результат
- меньшую цену ошибок
- меньше ручной боли
- лучший fit под мой GPU-сетап
- больше контроля над консистентностью персонажей и сцен

---

## Чего не делать

Не ставить целью сразу:
- полностью автоматическое “новелла -> готовая серия”
- длинное видео на каждую сцену
- идеальную авто-раскадровку без редакторского прохода
- один “магический” монолитный пайплайн, который делает всё сам

Почему:
- качество ломается именно на этапе **декомпозиции текста в киноязык**
- персонажи начинают “плыть”
- сцены распадаются по стилю
- длинное видео резко увеличивает стоимость времени и ошибок
- самый хрупкий участок — не рендер, а **story interpretation**

---

# Рекомендуемая стратегия

## Лучшая форма MVP

Сделать систему, которая превращает главу в:

1. список сцен  
2. список шотов внутри сцены  
3. JSON-манифест  
4. набор ключевых кадров  
5. по желанию — лёгкие анимированные вставки  
6. финальную склейку в формате:
   - PDF/CBZ-like comic
   - vertical storyboard
   - short motion comic video
