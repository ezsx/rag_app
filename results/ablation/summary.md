## Key Findings

### 1. Instruction Prefix — самый значимый одиночный параметр

Отключение instruction prefix даёт **+5.8% R@5** и **+3.6% MRR**.

Причина: evaluate_retrieval.py использовал prefix (`"Instruct: Given a user question..."`) при embedding запроса. Но в production pipeline (DEC-0042) pplx-embed работает **без prefix**. Eval скрипт тестировал несуществующую конфигурацию.

### 2. Dense limit — второй по влиянию параметр

| Dense limit | R@5 | Δ |
|------------|------|---|
| 10 | 0.817 | −0.017 |
| **20** (baseline) | **0.833** | — |
| **40** | **0.867** | **+0.034** |
| 60 (с combo) | 0.900* | — |

Больше dense-кандидатов → ColBERT получает больше разнообразных документов для rerank. Оптимум = 40 (diminishing returns после).

### 3. ColBERT — критически важен

Без ColBERT: R@1 падает с 0.708 до 0.467 (−34%), R@5 — с 0.833 до 0.733. ColBERT обеспечивает точную re-ранжировку на уровне токенов, что критично для top-K quality.

Latency: ColBERT добавляет ~2.2s/query (4.35s vs 2.18s) — это 100% overhead, но quality оправдывает.

### 4. RRF weights — не влияют при наличии ColBERT

Веса [1:1], [1:2], [1:3] — **идентичные результаты**. ColBERT rerank стирает различия в RRF ordering. Веса [1:4] и [1:5] слегка хуже (слишком много BM25 push).

**Вывод**: RRF weights до ColBERT — не tuning target. Production [1:3] — fine.

### 5. BM25 limit — не влияет

BM25 limit 50, 100, 200 — **одинаковый результат**. BM25 top-100 уже покрывает все релевантные документы.

### 6. DBSF fusion — немного хуже RRF

DBSF: R@5 0.825 vs RRF 0.833 (−0.008). RRF предпочтительнее.

## Recommended Configuration

```
Pipeline: BM25(100) + Dense(40) → RRF [1.0, 3.0] → ColBERT MaxSim → dedup
Prefix: OFF (уже OFF в production)
```

**Что менять в коде:**
1. `src/adapters/search/hybrid_retriever.py` строка 257: `dense_prefetch_limit` → 40 (было 20)
2. `scripts/evaluate_retrieval.py` строка 27: уже добавлен `--no-prefix` flag

**Ожидаемый эффект (подтверждено):**
- R@1: 0.708 → 0.758 (+7.1%)
- R@5: 0.833 → 0.900 (+8.0%)
- R@20: 0.867 → 0.933 (+7.6%)
- MRR: 0.765 → 0.823 (+7.6%)
- Latency: без изменений (4.35s/query)

## Dataset Info

`datasets/eval_retrieval_v3.json`: 120 вопросов, 30 каналов, 6 категорий:
- factual (40%): конкретные вопросы о фактах из новостей
- temporal (15%): привязка к датам и периодам
- channel_specific (15%): вопросы с упоминанием конкретного канала
- comparative (10%): сравнения технологий/продуктов
- entity (10%): вопросы о конкретных организациях/людях
- edge (10%): сленг, разговорный стиль, неточные формулировки

Все вопросы — natural language (не copy-paste из документов). Валидация: 75.8% dense-only recall@20.
