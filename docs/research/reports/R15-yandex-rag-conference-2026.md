## Часть 1: RAG-системы (Андрей Соколов)

Андрей Соколов — NLP-инженер в Яндексе, занимается RAG для Алисы, нейросаппорта и нейроюриста. Доклад покрывает три продакшн RAG-системы Яндекса с разными требованиями.

### Что мы уже делаем и Яндекс подтверждает

**1. RAG нужен даже при длинных контекстах**
- **Яндекс**: Модели с 128K-1M контекстом хорошо решают needle-in-a-haystack, но при дистракторах качество резко падает. "От рака пока не уйти" (81:48).
- **Мы**: Используем RAG + ReAct вместо запихивания всех 13K документов в контекст. Подтверждено.

**2. Композитность системы — основная сложность**
- **Яндекс**: "Изменение одного элемента приводит к изменению другого" (83:00-83:10). Рефразер, ретривер, генератор — все связаны.
- **Мы**: Испытали на себе — entity inject сломал Q4/Q9, system prompt без tool names → 0/30 tool selection. Наш evaluation-driven подход (22 эксперимента в playbook) — правильный ответ на эту проблему.

**3. Агентный подход для сложных RAG**
- **Яндекс**: Нейроюрист — один агент как единая точка входа, множество тулов (поиск, рассуждения, анализ документов) (98:35-99:03). Обучение через RL с reward-моделями.
- **Мы**: ReAct агент с native function calling, 3 specialized search tools + rerank + compose_context. Та же архитектура, но без RL-обучения.

**4. Context engineering > prompting**
- **Яндекс**: Динамическая подкладка тулов — не все 1000 тулов в контекст, а отбор релевантных (100:36-101:06). "RAG для RAG" — рекурсивность.
- **Мы**: Dynamic tool visibility — скрываем temporal_search/channel_search по query signals. Hints injection в system prompt. Яндекс валидирует наш подход. "Less is More" из R13 — то же самое.

**5. Итеративный поиск (refinement)**
- **Яндекс**: "Этот поиск будет итеративным, мы там пару раз сходим в рефразер, потому что поймем, что достаем не очень релевантные чанки" (82:18-82:30).
- **Мы**: Coverage threshold 0.65 + max 2 refinements — ровно это.

### Новое из RAG-доклада

**6. Robustness-метрики (ВЫСОКИЙ ПРИОРИТЕТ)**

Яндекс (88:18-89:28) — три метрики которых у нас нет:
- **Non-degradation rate (NDR)**: добавление контекста не должно ухудшать качество. Если хуже → что-то сломано.
- **Retrieval size robustness (RSR)**: больше контекстов → монотонный рост качества, не падение.
- **Retrieval order robustness**: перестановка чанков не должна менять ответ (lost-in-the-middle).

Статус у нас: НЕ измеряем. Eval только recall@5 и coverage.
Рекомендация: Добавить в eval pipeline. Для RSR — тест с k=3,5,10,15. Для order — shuffle чанков.

**7. RAG necessity classifier (СРЕДНИЙ ПРИОРИТЕТ)**

Яндекс (91:00-92:52):
- Не всем запросам нужен RAG. Определяют через: перплексию, tool call решение модели, обратный сигнал (найти запросы где контекст ничего не меняет).
- Обучают быстрый классификатор → **до 25% экономии** на инференсе + рост качества.

Статус у нас: Нет. Агент всегда идёт в поиск (forced search).
Рекомендация: Начать с эвристики (conversational queries → skip RAG). Полноценный classifier — после накопления логов.

**8. SFT + RLHF для RAG-поведения**

Яндекс (96:20-97:10):
- SFT на ~10K примерах: хорошие чанки → ответ, плохие → отмороз, NDA → блокировка.
- RLHF с reward-моделями для отморозов и NDA-сливов.
- **+19% прирост** vs промтированные модели.

Статус у нас: Qwen3-30B-A3B без дообучения, только промтирование.

**9. RL для агентного поведения (RESEARCH)**

Яндекс (99:05-100:12):
- Environment с симуляцией пользователя.
- Отдельные rewards: правильность тулов, качество ответа, оптимальность трейса.
- Policy update через PPO/GRPO.

**10. Три стадии оценки качества (ВЫСОКИЙ)**

Яндекс (85:18-87:57):
1. Бенчмарки (multihop, атрибуция, подтверждённость) — но нельзя для end-to-end.
2. Асессорские замеры (полезность, фактологичность) — по отдельным критериям.
3. A/B эксперименты — end-to-end с реальными пользователями.

"Чем больше градусников, тем лучше" (93:23) — главный тейкавей.

## Часть 2: Мультиагентная система MarketEye (Владислав Вихров)

Владислав Вихров — 3 года в Яндексе, MarketEye — мультиагентная система для продавцов Яндекс.Маркета.

### Архитектура

- **LLM-диспетчер** → специализированные агенты (справка, аналитика, chat, fallback на оператора)
- **Structured output** на диспетчере — гарантирует фиксированный формат имён агентов
- **Метрики диспетчера**: 97% precision, 92% recall для аналитики; 82%/96% для справки
- **История диалога**: 3+ сообщения критичны для правильной классификации
- Агенты могут использовать друг друга как tool (текстовый ответ, без обмена состоянием)

### Справочный агент (RAG)

- **454 документа, 7M+ токенов** документации Маркета
- **Custom embedding** обученный на синтетических данных
- **Embedding TOP-1=0.42, TOP-30=0.76, TOP-50=0.80**
- **Coverage 99%** на golden dataset (500 пар вопрос-документ)

### Подготовка данных для RAG

1. Удаление мета-информации (фронтенд-разметка)
2. **Таблицы: Markdown → JSON** (модели лучше парсят JSON чем Markdown таблицы)
3. Разбиение на логические чанки по семантической структуре документов
4. Ежедневное переиндексирование

### Синтетическая генерация вопросов для embedding

- Берут случайную подстроку документа → LLM генерирует вопрос к ней
- На входе: подстрока + полный текст + путь в справке + few-shot примеры стиля
- Распределение синтетических вопросов соответствует реальным вопросам продавцов
- Используется для обучения custom embedding модели

### Tool Design (принципы от Anthropic)

- **Консолидация**: объединение 4+ API-ручек с одинаковыми входами в 1 tool
- **Семантические ID**: текстовые идентификаторы (категория) вместо цифр
- **Явное описание**: когда использовать tool, когда НЕ использовать
- **Разделение ответственности**: `business_id` — hardcode в сервисе, информативные параметры — генерирует агент
- **Терминология**: явные ключи ("выручка, рубли" вместо "value")
- **Пагинация**: обработка большого объёма данных порциями

### Evaluation

- **Golden dataset**: 500 пар (вопрос, релевантный документ), собран с операторами поддержки
- **Асессорская разметка**: несколько экспертов на один вопрос
- **Коэффициент согласия AC1** (учитывает ожидаемое согласие, лучше Cohen's Kappa)
- **Алгоритм David-Skene** (E-M family) для взвешивания разметчиков — умнее majority vote
- Пример ошибки: асессор пометил "20M РУБ" как верный, но вопрос был про доставки (19M) — нужна точная разметка критериев

### Что НЕ сработало

1. **LLM as Judge НЕ РАБОТАЕТ для фактической корректности** — судья видит те же данные что и агент, не может проверить галлюцинации
2. **Fine-tune диспетчера** — особого выигрыша не было, few-shot достаточно
3. **Документ-агент с переиндексацией** — качество растёт, но "сильно растут тайминги"

### Что применимо к нам

| Яндекс MarketEye | Наш rag_app | Вывод |
|---|---|---|
| LLM-диспетчер (97/92%) | query_planner + strategy routing | Аналогичная архитектура |
| Tool design от Anthropic | Tool descriptions с примерами | Уже делаем, подтверждено |
| Synthetic questions для embedding | — | **Можем расширить eval dataset** |
| Таблицы: Markdown→JSON | Telegram посты (plain text) | Неприменимо |
| AC1 + David-Skene для асессоров | LLM judge (Claude) | Стоит рассмотреть для consistency |
| **LLM as Judge fails** | LLM judge — наш основной eval | **⚠️ ПРЕДУПРЕЖДЕНИЕ** — ограничение |
| Golden dataset 500 пар | eval v3 30 вопросов | Нужно масштабировать |

**Критическое**: Яндекс говорит LLM as Judge не работает для фактической корректности. Наш Claude judge — лучше чем strict recall, но нужно осторожно: LLM judge хорош для оценки полезности/полноты, но может пропускать фактические ошибки. Нужны **multiple criteria**: отдельно полезность, отдельно фактологичность, отдельно подтверждённость цитатами.

## Часть 3: Function Calling с синтетикой (Ольга Цымбой, Рамиль Латыпов)

Команда Agentica Search, Яндекс. Улучшение FC способностей через синтетические данные.

### Проблема

- 57% компаний используют агентов в продакшене, но LLM падают на цепочках Function Calling
- 32B модели значительно хуже топовых на FC задачах
- На русском качество ещё ниже — модели неправильно переводят строки при вызове инструментов
- Существующих датасетов мало, они одношаговые (1 запрос → 1 вызов)
- Ручная разметка многошаговых сценариев невозможна из-за сложности

### Синтетический pipeline

**Шаг 1: Генерация инструментов**
1. Кластеризация + дедупликация открытых API-наборов
2. Self-instruct для генерации новых тематик (несколько итераций)
3. На каждую тематику генерируют инструменты
4. **Merge**: объединение семантически похожих инструментов в один сложный
5. **Refine**: усложнение описания, расширение аргументов и типов
6. **Compression**: сокращение описания с сохранением функциональности
7. **Результат**: 4 варианта каждого инструмента (simple → complex) — **управление сложностью**
8. **Итого: 45K инструментов** по тематикам

**Шаг 2: Генерация диалогов (мультиагентная симуляция)**

Наивный подход (одна модель генерирует вопрос и ответ) не работает:
- Вырождение и mode collapse
- Нет верификации корректности

Рабочий подход — **двухфазный**:

*Фаза 1 — Планирование*:
- Структурированный JSON: персона пользователя, цели, ожидаемые вызовы инструментов, результаты
- Фиксирует эталонный план → каждый шаг ассистента можно верифицировать

*Фаза 2 — Мультиагентная симуляция*:
- **User-агент**: видит только персону и цели, НЕ знает об инструментах ассистента
- **Assistant-агент**: видит весь контекст + список инструментов, решает вызвать tool или ответить текстом
- **Tool-агент**: валидирует вызовы (алгоритмический + семантический уровень), исполняет, возвращает результаты
- Каждый агент видит ровно то, что видел бы в реальности → реалистичные диалоги

*Оптимизация качества*:
- **Множественные попытки**: 3 для user, 5 для assistant (majority voting)
- **Двухуровневая валидация**: JSON-схема (алгоритм) + уместность действия (модель)
- **Обработка ошибок**: продолжают генерацию, помечают ошибки, маскируют loss — модель учится исправлять свои ошибки
- **Языки**: генерируют нативно на русском и английском, НЕ переводят

**Результат**: **1.5M тренировочных сэмплов** (1.2M EN + 0.3M RU)

### Ключевое открытие: сложность FC

**Сложность FC коррелирует НЕ с длиной диалога или количеством вызовов, а с суммарным размером tool descriptions в system prompt.**

- 2000 токенов описаний → все модели справляются
- Десятки тысяч токенов → даже сильные модели падают
- 4 варианта сложности инструментов (simple → complex) позволяют управлять этим при обучении

**Это подтверждает**: нашу dynamic tool visibility (скрывать ненужные тулы = уменьшать размер descriptions = повышать accuracy) и "Less is More" paper из R13.

### Результаты обучения

**SFT (Supervised Fine-Tuning)**:
- Модель: Qwen 32B
- Данные: **только синтетика**, без единого реального размеченного примера
- Балансировка: 50% FC вызовов + 50% текстовых ответов, 50% RU + 50% EN
- Результат: значимые улучшения, особенно на русском

**RL / GRPO**:
- Reward: для FC — сравнение с ground truth, для текста — проверка отсутствия FC
- **Проблема**: модель начала галлюцинировать вызовы инструментов даже где их нет
- **Решение — irrelevance subset**: нужный инструмент исключается из доступных → модель ДОЛЖНА дать текстовый ответ (не вызывать tool)
- Результат: равномерный прирост на всех бенчмарках

**Итоговая метрика**: T-Pro 2.1 (32B) **сопоставим с Claude 3.5 Sonnet** по FC при значительно меньшем размере.

FC данные не противоречат общему датамиксу и не деградируют другие способности модели.

### Что применимо к нам

| Яндекс FC | Наш rag_app | Вывод |
|---|---|---|
| Qwen 32B — тот же класс | Qwen3-30B-A3B (3B active) | Те же проблемы с FC |
| Размер descriptions = bottleneck | 3-7 tools в agent schema | Dynamic visibility ✅ подтверждено |
| Irrelevance subset | Forced search (всегда в поиск) | **Нужен skip-RAG** для conversational |
| 1.5M synthetic FC data | 30 eval questions | Масштабировать eval |
| SFT на синтетике = Claude-level | Промтирование only | Path к SFT если решим обучать |
| Нативная генерация на RU | RU Telegram каналы | Синтетика работает на русском ✅ |
| 4 уровня сложности tool descriptions | — | Можем варьировать detail в tool schemas |

**Ключевое для нас**:
1. **Dynamic tool visibility** — валидирована и Яндексом. Размер descriptions влияет на accuracy больше чем количество тулов.
2. **Irrelevance subset при обучении** = **RAG necessity classifier** из RAG-доклада. Оба говорят: модель должна уметь НЕ вызывать tools.
3. **Синтетика работает на русском** — если будем делать SFT для Qwen3, можно генерировать данные по их pipeline.

## Неприменимо к нашему проекту

- **NDA/безопасность утечек**: у нас открытые Telegram-каналы, нет закрытой документации.
- **Высоконагруженная система (TTFT оптимизация Алисы)**: у нас single-user, latency не критична.
- **Перплексия как сигнал**: требует доступ к logprobs, llama-server может не отдавать.
- **Аналитический агент с API Маркета**: у нас нет структурированных API, только текстовые документы.
- **Fine-tune диспетчера**: у нас нет отдельного диспетчера, routing встроен в query_plan.
