# Промпт для исследования методов оценки качества ReAct агента

## Контекст проекта

Ты получаешь техническую спецификацию ReAct агента, который реализован в системе RAG (Retrieval-Augmented Generation) для ответов на вопросы по коллекции документов из Telegram каналов.

### Архитектура агента

ReAct агент реализует пошаговый цикл мышления-действия-наблюдения (ReAct) с детерминированной логикой:

1. **Цикл ReAct**: Thought → Action → Observation → FinalAnswer
2. **7 базовых инструментов**:
   - `router_select` - выбор маршрута поиска (bm25/dense/hybrid)
   - `query_plan` - планирование запросов с декомпозицией
   - `search` - гибридный поиск (BM25 + dense embeddings) с RRF слиянием
   - `rerank` - переранжирование через кросс-энкодер
   - `fetch_docs` - получение документов по ID
   - `compose_context` - сбор контекста с цитированием
   - `verify` - верификация утверждений через поиск

3. **Детерминированная логика**:
   - Автоматическая проверка **citation coverage** (≥80%) после `compose_context`
   - **Refinement раунды** (≤1) при недостаточном покрытии - дополнительный поиск с расширенными параметрами
   - **Верификация** финального ответа через `verify` инструмент (confidence ≥0.6)

4. **Метрики агента**:
   - **Citation Coverage**: `len(citations) / len(docs)` - доля документов, включенных в контекст
   - **Verification Confidence**: средняя уверенность по документам (0.0-1.0)
   - **Latency**: время выполнения от запроса до финального ответа
   - **Refinement Count**: количество выполненных refinement раундов

5. **SSE стриминг**: Агент отправляет события (`step_started`, `thought`, `tool_invoked`, `observation`, `citations`, `final`) для наблюдаемости

### Данные и источники

**Формат данных**:
- **ChromaDB** (dense embeddings): векторное хранилище с эмбеддингами через `multilingual-e5-large`
- **Tantivy/BM25** (sparse search): полнотекстовый индекс для BM25 поиска
- **Гибридный поиск**: RRF (Reciprocal Rank Fusion) слияние результатов BM25 и dense поиска

**Источники данных**:
- Telegram каналы (мировые новости, технические новости, и т.д.)
- Каждое сообщение индексируется с метаданными:
  - `channel_id`, `channel` - идентификатор и название канала
  - `message_id` - ID сообщения в Telegram
  - `date` - дата сообщения (YYYY-MM-DDTHH:MM:SSZ)
  - `author` - автор сообщения (если есть)
  - `text` - текст сообщения
  - Дополнительные метаданные (forward, reply, и т.д.)

**Примеры запросов**:
- "Какие важные объявления делал @durov в августе 2023?"
фактуальные вопросы с временными фильтрами
- "Что обсуждали в канале @news о криптовалюте?"
поиск по теме в конкретном канале
- "Расскажи о новых функциях Telegram за последний месяц"
временные запросы
- "Что писал @username о проекте Y?"
поиск по автору и теме
- "Какие изменения произошли в политике канала Z?"
комплексные вопросы, требующие множественных шагов

### Текущее состояние оценки

**Что уже есть**:
- Базовый датасет `datasets/eval_questions.json` с 8 вопросами разных категорий
- Симуляционный скрипт `tests_minimal/test_agent_dataset.py` (только логика, не реальные запросы)
- План оценки в документации с метриками: Answer Accuracy/F1, Faithfulness (Citation Precision), Coverage/Recall, Conciseness, Latency

**Что отсутствует**:
- Полноценный скрипт для автоматической оценки агента на реальных запросах
- Автоматический расчет метрик (Citation Precision, Recall@5, Latency)
- Сравнение результатов baseline QA vs Agent
- Go/no-go критерии для принятия решений

## Задача исследования

Необходимо исследовать и предложить решение для **комплексной оценки качества ReAct агента** на реальных запросах из Telegram данных.

### Цели оценки

1. **Количественная оценка качества ответов**:
   - Точность ответов (Answer Accuracy/F1)
   - Верифицируемость (Faithfulness / Citation Precision)
   - Покрытие релевантной информации (Coverage/Recall)
   - Латенси и производительность

2. **Сравнение с baseline**:
   - Сравнение Agent vs классический QA сервис (`QAService`)
   - Определение улучшений для сложных запросов
   - Выявление регрессий на простых запросах

3. **Диагностика проблем**:
   - Анализ причин неудачных ответов (ошибки инструментов, планирования, верификации)
   - Выявление паттернов ошибок по категориям запросов

4. **Принятие решений**:
   - Go/no-go критерии для включения агента в production
   - Метрики для мониторинга качества в реальном времени

### Ограничения и требования

**Технические**:
- Агент работает через SSE стриминг (`/v1/agent/stream`)
- Baseline QA доступен через `/v1/qa` endpoint
- Данные в ChromaDB и BM25 индексе (Telegram сообщения)
- Оценка должна работать автоматически (без ручной проверки каждого ответа, где возможно)

**Контекст данных**:
- Запросы могут требовать фильтрации по дате, каналу, автору
- Ответы должны содержать цитаты на источники из Telegram
- Релевантность может зависеть от временного контекста (например, "последний месяц")
- Некоторые запросы могут не иметь ответа в базе (отрицательные кейсы)

**Метрики из плана**:
- **Answer Accuracy/F1**: Содержит ли ответ правильную информацию? (может потребовать ручную проверку для сложных случаев)
- **Faithfulness (Citation Precision)**: Доля утверждений в ответе, которые подтверждены источниками (идеально 100%)
- **Coverage/Recall**: Использована ли большая часть релевантной информации из источников? Был ли нужный документ в топ-5 результатов поиска?
- **Conciseness & Correctness**: Длина ответа vs ожидаемая, нет ли избыточной информации
- **Latency**: Среднее и P95 время выполнения vs baseline QA

**Success Criteria** (из плана):
- Агент отвечает корректно хотя бы на **70% вопросов** eval набора
- Baseline может отвечать на ~50% сложных вопросов
- Для 80% запросов агент должен иметь полную поддержку ответа (coverage ~100%)
- Если baseline пропустил ответ, агент должен его найти
- Типичный запрос агента < 10 секунд, практически все < 30 секунд
