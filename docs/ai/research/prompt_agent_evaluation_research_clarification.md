# Уточнение по формату результата исследования

## Ответ на уточняющий вопрос

**Формат ожидаемого результата**: Структурированный исследовательский документ в текстовом формате по указанной структуре, **БЕЗ** полноценных файлов кода, но **С** примерами структур данных и псевдокодом там, где это необходимо для понимания.

### Что нужно:

1. **Исследовательский документ** (основной результат):
   - Структурированный ответ по формату из промпта
   - Методология оценки с обоснованием выбора метрик
   - Архитектура решения с деталями реализации
   - Спецификация метрик с формулами и алгоритмами расчета
   - План реализации с приоритетами

2. **Примеры структур данных** (не код, а схемы):
   - JSON схема для расширенного датасета (структура полей, типы данных, примеры значений)
   - Структура отчета оценки (какие поля должны быть, формат данных)
   - Примеры входных/выходных данных для метрик

3. **Псевдокод/алгоритмы** (где критично для понимания):
   - Алгоритмы расчета сложных метрик (например, Citation Precision)
   - Логика сравнения Agent vs Baseline
   - Пошаговый процесс оценки

4. **Ссылки и источники**:
   - Актуальные исследования и статьи (последние 2-3 года)
   - Промышленные практики (AWS Bedrock, LangChain Evaluation, и т.д.)
   - Обоснование выбора подходов

### Что НЕ нужно:

- ❌ Полноценные Python скрипты (мы напишем сами на основе спецификации)
- ❌ Jupyter ноутбуки
- ❌ Готовые библиотеки/фреймворки для использования (достаточно названий и ссылок)
- ❌ Большие блоки кода (только минимальные примеры структур/алгоритмов)

### Фокус исследования:

**Главная цель**: Получить достаточно деталей для написания **технической спецификации** и **плана реализации** инструмента оценки.

**Критично понять**:
- **ЧТО** измерять (какие метрики, почему именно эти)
- **КАК** измерять (методы расчета, автоматизация vs ручная проверка)
- **КАК** структурировать (формат датасета, формат отчета, архитектура скрипта)
- **ПОЧЕМУ** выбран именно такой подход (обоснование из исследований)

**Примеры деталей, которые нужны**:
- Формула Citation Precision: как именно считать утверждения в ответе? Как сопоставлять с источниками?
- Recall@5: как определять "нужный документ"? По ID? По содержанию?
- Автоматизация Faithfulness: можно ли использовать LLM-as-judge? Какие промпты? Какие метрики качества судьи?
- Структура датасета: какие поля критичны? Как помечать отрицательные кейсы? Нужны ли эталонные ответы?

### Формат примеров в документе:

Вместо кода использовать:
- **JSON схемы** для структур данных
- **Псевдокод** для алгоритмов (высокоуровневые шаги)
- **Таблицы** для сравнения подходов
- **Формулы** для метрик
- **Диаграммы** (текстовые) для архитектуры

### Примеры того, что достаточно:

**Вместо Python функции:**
```
Алгоритм расчета Citation Precision:
1. Разбить ответ на утверждения (предложения или семантические единицы)
2. Для каждого утверждения найти соответствующий источник в citations
3. Если источник найден и подтверждает утверждение → supported = True
4. Precision = count(supported=True) / count(all_statements)
```

**Вместо полного скрипта:**
```
Процесс оценки для одного вопроса:
1. Загрузить вопрос из датасета
2. Вызвать Agent API (/v1/agent/stream), собрать все события
3. Вызвать Baseline QA API (/v1/qa), получить ответ
4. Извлечь метрики из ответов (coverage, citations, latency)
5. Применить дополнительные метрики (Citation Precision, Recall@5)
6. Сохранить результаты в структурированный формат
7. Повторить для всех вопросов
8. Сгенерировать сравнительный отчет
```

## Итоговый формат ответа

**Основной документ**: Детальное исследование по структуре из промпта с:
- Методологией и обоснованием
- Архитектурой решения
- Спецификацией метрик
- Планом реализации

**Дополнительно**:
- JSON схемы для структур данных
- Псевдокод для критичных алгоритмов
- Ссылки на актуальные исследования и практики
- Примеры структур данных (не код)

**Результат должен позволить**:
- Написать техническую спецификацию инструмента оценки
- Спланировать реализацию по этапам
- Понять, какие метрики реализовать первыми (MVP)
- Принять обоснованные решения по архитектуре

---

**Важно**: Фокус на методологии, архитектуре и спецификации, а не на готовом коде. Код мы напишем сами на основе полученной спецификации.
